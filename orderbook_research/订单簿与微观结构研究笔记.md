### 0.1 两种通信模式

- 我觉得做订单簿研究，本质上是在和交易所的服务器实时通信，然后在本地维护一份订单薄数据；但通讯有两种完全不同的通信模式：

**HTTP（请求-响应模式），即轮询**


```
（客户端）          交易所服务器
     ──── 发起请求 ────>
    <──── 返回数据 ────
         （连接关闭）
```

- 客户端问一次，服务器就答一次，答完之后断开连接；适合"需要某个数据，拿到就走"的场景。

- 对应后续订单簿研究里的：拉取一次完整的订单簿快照。

**WebSocket（持久连接模式）**

```
（客户端）          交易所服务器
     ──── 握手请求 ────>
    <──── 握手确认 ────  ！！！！
         （连接保持）
    <──── 推送消息 ────  （服务器主动推）
    <──── 推送消息 ────
    <──── 推送消息 ────
    ...（直到主动断开）
```

- 一次握手成功后，连接会一直保持，服务器可以随时往客户端推送消息，而不需要客户端反复请求；适合"需要持续接收实时数据"的场景。

- 对应后续订单簿研究里的：持续接收订单簿的每次变化。

**URL 前缀的区别**

```
HTTP    → http://  或  https://（加密）
WebSocket → ws://   或  wss://（加密）

币安 REST 接口：  https://fapi.binance.com/fapi/v1/depth
币安 WS 接口：    wss://fstream.binance.com/ws/btcusdt@depth
```

---

### 0.2 REST 和 WebSocket 在重建订单簿时的分工协作

- 两者不是竞争关系，而是互补的：

```
某次启动时：
  REST（HTTP）拉一次完整的订单簿快照
        ↓
之后持续：
  WebSocket 持续接收每一次订单簿的变化（差量）
        ↓
本地维护：
  把快照 + 所有差量合并，得到实时的完整订单簿
```

总的来说只用REST反复轮询是可以的，但有延迟，而且频率受限；而只用WebSocket又不行，因为它只实时推送变化，但却没有初始完整状态；所以把两者结合，得到准确的实时订单簿。

### 0.3 Python 里的对应库及其用法
- HTTP、HTTPS 是【底层传输协议】，REST 是【设计风格 / 接口规范】

- REST全称Representational State Transfer（表述性状态转移），是一套设计 API 接口的风格 / 规范，基于 HTTP/HTTPS 工作，它规定了：
  
```
用 GET 获取数据
用 POST 提交数据
用 PUT 修改数据
用 DELETE 删除数据
URL 要简洁清晰
```


| 通信模式 | Python 库 | 用途 |
|---|---|---|
| HTTP /HTTPS/ REST | `requests` | 拉取一次性快照，发送 API 请求 |
| WebSocket | `websocket-client` | 建立持久连接，持续接收推送消息 |


**requests**

```python
import requests
resp = requests.get("https://fapi.binance.com/fapi/v1/depth?symbol=BTCUSDT&limit=1000")
data = resp.json()   # 得到完整订单簿快照
```

- 等价于我们在浏览器地址栏输入那个网址然后拿到 JSON。（客户端 ——请求——> 服务器）

**websocket-client**

```python
import websocket

def on_message(ws, message):
    print(message)   # 每次收到推送都调用这里

ws = websocket.WebSocketApp(
    "wss://fstream.binance.com/ws/btcusdt@depth",
    on_message=on_message,
)
ws.run_forever()   # 连接保持，持续接收
```

- `run_forever()` 会阻塞当前线程一直监听消息，所以实际使用时把它放在独立线程里，主线程负责处理收到的消息。

---

### 0.4 线程与队列数据结构

- 因为 WebSocket 的 `run_forever()` 会占用一整个线程，而主线程还需要处理消息、计算指标，所以需要用多线程 + 队列的模式：

```
后台线程（websocket-client）
  持续接收 WS 推送 → 把消息放入 queue.Queue

主线程
  从 queue.Queue 取出消息 → 更新订单簿 → 计算指标 → 打印
```

- `queue.Queue` 是 Python 标准库里的线程安全队列，专门用于线程间传递数据，不需要手动加锁。

## 1. “宏观结构”K线数据 vs “微观结构”订单簿数据

| 对比项 | K线数据 | 订单簿数据 |
|---|---|---|
| 时间粒度 | 分钟级、小时级、日、周、月级别 | 毫秒级、微秒级 |
| 信息来源 | 已成交的交易汇总 | 当前所有未成交挂单 |
| 每条记录 | open / high / low / close / volume | L2: price / qty |
| 本质 | 过去一段时间成交的压缩快照 | 当前市场的"意图快照" |
| 缺点 | 信息已压缩，无法还原个别成交；看不到当前挂单和买卖力量对比 | 变化极快，数据量极大；挂单可随时撤，不一定代表真实成交意愿 |

对比来看二者本质上都是离散型时间序列数据，但也存在一些不同点：

```
K线 = 过去发生了什么
订单簿 = 现在大家打算做什么
```

数据精度从粗到细：

```
K线（压缩汇总）← Tick Trades（逐笔成交）← L2 订单簿（挂单意图）
```

## 2. 订单簿上的挂单 vs Tick Trades

**挂单（Order Book 里的数据）**

- 含义：某人在某价格挂了一笔限价单，还没成交
- 状态：pending，随时可以撤
- 不代表真实成交意愿（可以是刷单、幌骗 Spoofing）

**Tick Trades（逐笔成交数据）**

- 含义：一笔买单和一笔卖单真实撮合完成
- 状态：已发生，不可撤销
- 每条记录包含：price / qty / side（主动方向，T or M）/ timestamp


两者的关系：

```
挂单进入订单簿 → 等待对手方 → 撮合成功 → 从订单簿消失 → 生成一条 Tick Trade

也就是说：Tick = 一笔真实成交 Tick Trades = 市场上真实发生的每一笔成交记录 K线 = 把一堆 Tick Trades 打包、压缩、统计后的结果
```

---

## 3. 交易所的撮合方式

### 3.1 价格优先、时间优先（Price-Time Priority）

这是我了解到的最常见的撮合方式，币安也采用此方式。

```
规则：
  不同价格：最优价格（买方最高价 / 卖方最低价）先成交
  同等价格：先挂先成交

例子：
  卖A：price=100，time=09:00:01
  卖B：price=100，time=09:00:02
  卖C：price=101，time=09:00:00

  买单 price=101，qty=1 进来 → 先匹配卖A（价格最低，且比卖B早）
```

### 3.2 价格优先、量大优先（Pro-Rata）

同等价格下，挂单量大的优先成交，而不是先挂先得。
用于部分期权、期货交易所，鼓励大单提供流动性。

### 3.3 Maker / Taker 的本质

```
限价单（Limit Order）→ 挂到订单簿等待 → Maker（提供流动性）
市价单（Market Order）→ 立刻吃掉最优挂单 → Taker（消耗流动性）
止损单（Stop Order）→ 触发条件后转为市价单执行
```

币安手续费 Maker < Taker，因为 Maker 对市场流动性有贡献。

## 4. L1 / L2 / L3 订单簿分别是什么

### L1 订单簿（最佳买卖价）

```
只包含：best bid【最高买价（最优买家，即订单薄上出价最高的人） + 数量】和 best ask【最低卖价（最优卖家，即订单薄上愿意接受的出售价最低的人） + 数量】
用途：快速判断当前价格，计算 spread
```

字段示例：

```
bid: 30000.00 × 1.5
ask: 30000.50 × 0.8
```

### L2 订单簿（聚合深度）

```
包含：多个价格档位，每档聚合所有挂单总量（5档、10档、20档、500档等）
用途：分析订单簿深度、流动性、价格压力
币安 /depth 接口返回的就是 L2 数据
注意：L2 是聚合的，所以只能看到某价格上共有多少量，但看不到是几个人挂的
```

示例：

```
asks:
  30001.00 × 2.5
  30000.80 × 1.2
  30000.50 × 0.8   ← best ask

bids:
  30000.00 × 1.5   ← best bid
  29999.80 × 3.0
  29999.50 × 4.2
```

### L3 订单簿（逐笔挂单）

```
包含：每一笔挂单的详细信息（order_id / price / qty / side / timestamp）
用途：追踪单个订单的生命周期（挂单→修改→撤单→成交）
币安不提供 L3 数据，其他地方提供但是需要付费
```

### 币安实际提供的数据级别

| 级别 | 接口 |
|---|---|
| L1 | ticker（best bid/ask） |
| L2 | depth（聚合深度快照 + 增量流） |
| Tick Trades | aggTrades / trades（逐笔成交） |
| L3 | 不开放 |

## 5. 深度（Depth）的含义

深度就是 L2 订单簿的价格档位层数，即能看到多少个价格级别。

**深度好（流动性好）**：

```
距离 best ask 稍远的价格上仍有大量挂单
即使大单进来，价格也不会滑太多
```

**深度差（流动性差）**：

```
ask: 30000.50 × 0.1
     30001.50 × 0.2    ← 中间有很大空洞
     30005.00 × 0.5

一笔大买单会直接把价格从 30000.50 推到 30005.00 → 滑点大
```

## 6. 币安订单簿的获取方式

### 6.1 REST 快照（Snapshot）

一次性拉取当前完整状态：

```
GET /fapi/v1/depth?symbol=BTCUSDT&limit=500

返回：
{
  "lastUpdateId": 1234567890,   // 当前版本号，用于增量流对齐
  "bids": [["30000.50", "1.200"], ...],
  "asks": [["30001.00", "0.500"], ...]
}
```

### 6.2 WebSocket 增量流（Diff Depth Stream）

订阅后实时推送每次变化（差量）：

```
wss://fstream.binance.com/ws/btcusdt@depth

每条推送字段：
  e: "depthUpdate"   事件类型
  E: 时间戳（毫秒）
  s: "BTCUSDT"
  U: 本条更新起始版本号
  u: 本条更新结束版本号
  pu: 上一条更新结束版本号（用于验证连续性）
  b: bids 变化列表，[price, qty]，qty=0 表示删除该档位
  a: asks 变化列表，同上
```

### 6.3 WebSocket 全量推送（Partial Book Depth）

直接推最优 N 档完整快照，不需要自己维护本地状态：

```
wss://fstream.binance.com/ws/btcusdt@depth20@100ms
每 100ms 推一次最优 20 档的完整快照
最简单，但深度有限（最多 20 档）
```

## 7. 重建本地订单簿的正确流程

用增量流维护本地订单簿的解决步骤（我用的是币安交易所）：

```
1. 订阅 WebSocket 增量流，开始缓存收到的消息（先不处理）
2. 用 REST 拉一份完整快照，记下 lastUpdateId = N
3. 丢弃所有 u <= N 的缓存消息
4. 找到第一条满足 U <= N+1 <= u 的消息，从它开始处理
5. 此后每条消息：
     验证 pu == 上一条的 u（版本连续性）
     遍历 bids/asks 差量：qty=0 删除，qty>0 新增或覆盖
6. 如果发现版本号不连续，重新走步骤 1-4
```

更新规则：

```
qty = 0  → 删除该价格档位
qty > 0  → 新增或覆盖该价格档位的数量
```

## 8. 流动性变化识别指标

### 8.1 买卖价差（Spread）

```python
spread = best_ask - best_bid
spread_bps = spread / mid_price * 10000   # 基点，更直观

信号：spread 突然扩大 → 流动性变差，市场紧张
```

### 8.2 最优 N 档深度（Order Book Depth）

```python
bid_depth = sum(qty for price, qty in bids[:N])
ask_depth = sum(qty for price, qty in asks[:N])

信号：买方挂单量突然减少 → 买盘撤退
```

### 8.3 买卖量不平衡（OBI，Order Book Imbalance）

```python
OBI = bid_depth / (bid_depth + ask_depth)

OBI > 0.6 → 买压明显，可能上涨
OBI < 0.4 → 卖压明显，可能下跌
OBI ≈ 0.5 → 买卖均衡
```

### 8.4 加权中间价（Weighted Mid Price）

```python
mid_price = (best_ask + best_bid) / 2
weighted_mid = (best_bid_price * best_ask_qty + best_ask_price * best_bid_qty) / (best_bid_qty + best_ask_qty)

weighted_mid < mid_price → 买方对价格压制，短期偏空（买压）
weighted_mid > mid_price → 卖方对价格抬升，短期偏多（卖压）
```

逻辑：用对手方的量对本方最优价加权，买方量大则拉低加权中间价。

### 8.5 大单撤单检测

```
某价格档位上一个时刻有 50 BTC，下一时刻变为 0
→ 大单撤退，流动性真空，注意价格可能被推动
```

## 9. 交易数据的整体体系总结

```
参与者挂限价单
      ↓
进入订单簿（L2: 聚合视图 / L3: 逐笔视图）
      ↓
市价单吃单
      ↓
撮合引擎（价格优先、时间优先）
      ↓
产生 Tick Trade（逐笔成交）
      ↓
多根 Tick Trades 按时间窗口压缩
      ↓
K线（OHLCV）

数据精度：Tick Trades > L2 Order Book > K线
时间粒度：微秒          毫秒            分钟
信息内容：成交结果       挂单意图         压缩汇总
```
