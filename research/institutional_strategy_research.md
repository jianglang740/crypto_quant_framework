# Institutional Strategy Research Notes

这份文档记录当前阶段值得重点研究的五类机构化量化策略方向：因子投资、多因子选币、波动率策略、风险平价/波动率目标、多策略组合。

这些方向不是立即进入实盘的策略，而是后续研究路线。成熟结果应回到现有框架，通过数据、回测、入库、dashboard 观察和测试网验证。

## 1. 因子投资 / Smart Beta

### 核心思想

因子投资不是直接预测下一根 K 线涨跌，而是寻找长期有解释力的收益来源。传统市场常见因子包括：

- 动量：近期强势资产未来一段时间继续强势。
- 价值：价格相对基本面偏低的资产长期可能有溢价。
- 质量：盈利、现金流、资产负债更健康的公司更稳健。
- 低波动：低风险资产可能获得超额风险调整收益。
- 规模：小市值资产可能有风险溢价。
- Carry：持有资产本身带来的收益，例如利差、期限结构、资金费率。
- 流动性：低流动性资产要求更高风险补偿。

在加密市场里，传统财务因子很难直接搬过来，但可以形成加密版本：

- 价格动量因子
- 成交量变化因子
- 波动率因子
- 流动性因子
- 资金费率因子
- 基差因子
- 市值/成交额因子
- 链上活跃度因子
- 持仓集中度因子

### 适合我们当前框架的切入点

当前框架已经具备 K 线数据、回测、入库、dashboard，因此第一阶段适合先做 K 线可计算因子：

1. 动量因子
   - 最近 N 根收益率
   - 多周期收益率
   - 突破强度

2. 波动率因子
   - realized volatility
   - high-low range volatility
   - ATR 类波动

3. 成交量/流动性因子
   - 成交量变化率
   - 成交量分位数
   - 成交额近似值：close * volume

4. 回撤/风险因子
   - 最近 N 根最大回撤
   - 下行波动率
   - 收益偏度近似

### 数据需求

第一阶段只需要：

- `klines.open`
- `klines.high`
- `klines.low`
- `klines.close`
- `klines.volume`
- `symbol`
- `timeframe`
- `exchange`
- `open_time`

第二阶段再扩展：

- funding rate
- open interest
- market cap
- 链上数据
- 多交易所价格

### 建议研究脚本

后续可新增：

- `research/scripts/factor_observation.py`

职责：

- 从 MySQL 读取 K 线。
- 计算基础因子。
- 观察因子分布、分位数、稳定性。
- 不直接做实盘，不直接下单。

### 当前优先级

高。

原因：因子是多因子选币、多策略组合、风险模型的基础。即使暂时只做 BTC 单标的，也可以先研究“时间序列因子”。等有多币种数据后，再扩展成横截面因子。

## 2. 多因子选币 / 多因子选股

### 核心思想

多因子模型是把多个因子组合成一个评分系统，然后按分数构建组合。

典型流程：

```text
收集资产池数据
→ 计算因子
→ 标准化
→ 去极值
→ 缺失值处理
→ 因子加权
→ 资产排序
→ 组合构建
→ 定期调仓
→ 回测和归因
```

在加密市场里，多因子选币可以用于：

- 多币种现货轮动
- 多币种合约多空
- 强势币选择
- 弱势币过滤
- 行情阶段切换时的资产筛选

### 适合我们当前框架的切入点

当前只有 BTC/USDT 数据时，不适合真正做横截面多因子。多因子需要一个资产池，例如：

- BTC/USDT
- ETH/USDT
- SOL/USDT
- BNB/USDT
- XRP/USDT
- DOGE/USDT
- ADA/USDT
- AVAX/USDT
- LINK/USDT
- LTC/USDT

第一阶段应该先把多币种 K 线入库能力补起来，而不是马上写复杂模型。

### 数据需求

最低需求：

- 多币种同周期 K 线
- 时间对齐
- 成交量
- 足够长历史

增强需求：

- funding rate
- open interest
- 市值
- 交易所流动性
- 链上数据

### 实现路径

1. 数据准备
   - 扩展 K 线导入脚本，让多个 symbol 可以入库。
   - 确认每个 symbol 的 timeframe 一致。
   - 检查缺失 K 线。

2. 因子计算
   - 先实现价格、波动、成交量三类基础因子。
   - 每个时间点对所有币种做横截面排名。

3. 组合构建
   - 只做多：买排名靠前的币。
   - 多空：做多排名靠前，做空排名靠后。
   - 等权或按波动率倒数加权。

4. 调仓机制
   - 5m 级别不宜频繁换币，手续费会很重。
   - 可以先研究 1h、4h、1d 周期。

5. 回测入库
   - 需要组合级回测能力。
   - 如果暂时不用组合引擎，可以先研究信号有效性，再决定是否进入框架策略。

### 当前优先级

中。

原因：方向重要，但前置条件是多币种数据。当前框架更适合先把单标的研究链路打稳，再扩展到多币种。

## 3. 波动率策略

### 核心思想

波动率策略研究的不是单纯方向，而是市场波动状态。传统机构常见波动率策略包括：

- 隐含波动率 vs 实现波动率
- 波动率套利
- gamma scalping
- variance risk premium
- volatility targeting
- 波动率突破
- 波动率 regime detection

加密市场如果暂时没有期权数据，可以先研究 realized volatility，也就是由 K 线计算出来的历史波动。

### 适合我们当前框架的切入点

第一阶段适合做三类 K 线级波动率研究：

1. 波动率过滤器
   - 高波动时降低仓位。
   - 低波动时允许突破策略开仓。
   - 极端波动时停止交易。

2. 波动率突破
   - 波动压缩后突破。
   - ATR 上升确认趋势。
   - high-low range 扩张确认行情启动。

3. 波动率目标仓位
   - 目标不是预测方向，而是让每次交易风险接近。
   - 仓位 = 目标风险 / 当前波动率。

### 数据需求

第一阶段：

- OHLCV K 线

第二阶段：

- 期权 implied volatility
- funding rate
- open interest
- order book depth

### 当前已有切入点

`research/candidate_strategies.py` 中的 `ResearchVolatilitySqueezeBreakoutStrategy` 已经属于波动率方向的雏形：

- 用最近 K 线平均 range 判断波动压缩。
- 用前高突破判断方向。
- 用成交量放大做确认。
- 用止损、止盈、最长持仓控制风险。

后续可以演进成：

- ATR 版本
- 分位数波动率版本
- regime filter 版本
- volatility targeting position sizing 版本

### 当前优先级

高。

原因：波动率可以作为单独策略，也可以作为所有策略的风控层。对加密市场尤其重要，因为加密市场波动状态切换很快。

## 4. 风险平价 / 波动率目标 / 组合配置

### 核心思想

这类策略重点不是寻找单个交易信号，而是控制不同资产或策略之间的风险分配。

常见形式：

- 风险平价：每个资产贡献相近风险。
- 波动率目标：组合总波动率维持在目标附近。
- 动态降杠杆：波动变大时降低仓位。
- 最大回撤控制：回撤过大时减仓。
- 相关性约束：避免多个资产在危机时一起暴露。

### 适合我们当前框架的切入点

当前还不适合马上做完整风险平价，因为需要：

- 多资产数据
- 组合回测
- 相关性矩阵
- 组合级风控和归因

但可以先做“单策略波动率目标仓位”：

```text
计算最近 N 根 realized volatility
→ 设定目标风险
→ 当前波动越大，仓位越小
→ 当前波动越小，仓位可适当提高
```

这样即使只有 BTC/USDT，也能先研究 volatility targeting。

### 数据需求

第一阶段：

- 单资产 K 线
- 策略权益曲线

第二阶段：

- 多资产 K 线
- 多策略权益曲线
- 相关性矩阵
- portfolio-level backtest result

### 实现路径

1. 单策略层
   - 给候选策略增加 volatility-adjusted position sizing。
   - 对比固定仓位 vs 波动率目标仓位。

2. 单资产多策略层
   - 同一个 BTC/USDT 上跑多个策略。
   - 观察策略收益相关性。
   - 按波动率和回撤调整各策略权重。

3. 多资产组合层
   - 等有多币种数据后，再做真正风险平价。

### 当前优先级

中高。

原因：完整风险平价还早，但波动率目标仓位很适合现在开始做。它能直接改善单策略风控质量。

## 5. 多策略组合 / Multi-strategy

### 核心思想

机构很少只依赖单个策略。更常见的是多个低相关策略组合：

```text
趋势策略
+ 反转策略
+ 波动率策略
+ carry 策略
+ 套利策略
+ 多因子策略
+ 执行优化
```

多策略组合的关键不是让每个策略都极强，而是让它们：

- 收益来源不同
- 相关性低
- 回撤不同时发生
- 总体资金曲线更平滑
- 单策略失效时不会拖垮整体

### 适合我们当前框架的切入点

当前可以先做“研究层多策略比较”，暂时不急着做真正组合执行。

第一阶段：

- 每个策略单独回测。
- 把每个策略的 equity curve 入库。
- 在 research 中读取多个回测结果。
- 计算策略间相关性、回撤重叠、收益互补性。

第二阶段：

- 设计简单策略权重。
- 等权组合。
- 按波动率倒数加权。
- 按最大回撤惩罚加权。

第三阶段：

- 再考虑组合引擎和实盘运行方式。

### 数据需求

第一阶段：

- 多个单策略回测结果
- equity_curve
- trades
- strategy_runs.config

第二阶段：

- 统一时间轴的策略收益序列
- 策略间相关性
- 组合权益曲线

### 当前优先级

中高。

原因：你现在的框架已经能保存多个回测 run，dashboard 也能观察结果。先做研究层多策略比较很自然，但不要过早进入复杂组合引擎重构。

## 推荐研究优先级

结合当前框架状态，建议顺序是：

```text
1. 波动率策略
2. 因子投资的 K 线基础因子
3. 单策略波动率目标仓位
4. 多策略比较和相关性分析
5. 多币种多因子选币
6. 完整风险平价和组合配置
```

原因：

- 波动率和 K 线基础因子可以直接使用现有 BTC/USDT 5m 数据。
- 波动率目标仓位能直接改善策略风控。
- 多策略比较可以复用已有回测入库结果。
- 多因子选币和完整风险平价需要多币种数据，应该放在数据层扩展之后。

## 研究边界

- 不在 research 中放 API key、数据库密码、私钥。
- 不在 research 中做实盘下单。
- 不从 research 绕过现有回测引擎。
- 不把单次回测最优结果直接视为可实盘策略。
- 不为了提高回测收益无限扩大参数空间。
- 成熟策略必须迁回框架正式策略链路，并经过回测入库、dashboard 观察和测试网验证。
